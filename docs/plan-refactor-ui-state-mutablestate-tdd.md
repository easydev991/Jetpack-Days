# План рефактора UI-state: отказ от MutableState (TDD)

Дата: 2026-04-27  
Статус: Planned

## 1. Цель

- [ ] Убрать `MutableState<T>` из data-классов UI-state во всех экранах.
- [ ] Перенести мутабельность в ViewModel (`StateFlow`) и/или в Composable (`remember/rememberSaveable`).
- [ ] Сохранить текущее поведение приложения без регрессий, включая напоминания, бэкапы и навигацию.

## 2. Область изменений

- [ ] `ui/state/*` и state-модели внутри `ui/screens/*`.
- [ ] `ui/viewmodel/*` (обновление контрактов состояния и событий).
- [ ] Compose-экраны, которые читают/изменяют UI-state.
- [ ] Unit/UI-тесты, завязанные на старый mutable-паттерн.

## 3. Ограничения и правила проекта

- [ ] Следовать TDD: Red -> Green -> Refactor в каждом инкременте.
- [ ] Не использовать `!!`, применять безопасную работу с nullable.
- [ ] Не менять модель резервных копий и формат сериализации backup.
- [ ] После каждого инкремента запускать `make format` и релевантные тесты.
- [ ] Перед финалом запустить полный `make test`.

## 4. Зависимости и порядок выполнения

- [ ] Сначала зафиксировать тестами текущее поведение экранов.
- [ ] Затем мигрировать экраны по одному (от меньшего риска к большему).
- [ ] После стабилизации каждого экрана переходить к следующему.
- [ ] Общие утилиты/базовые state-контракты менять только после пилотного экрана.

## 5. Инкременты (TDD)

## 5.1 Инкремент A: Базовая фиксация поведения (RED-first)

- [ ] Провести инвентаризацию всех UI-state с `MutableState<T>`.
- [ ] Для каждого затронутого экрана добавить/усилить unit-тесты ViewModel:
- [ ] инициализация состояния;
- [ ] изменение полей;
- [ ] валидация;
- [ ] `hasChanges`/кнопка сохранения;
- [ ] ошибки и recovery.
- [ ] Добавить/актуализировать UI-тесты критичных сценариев:
- [ ] Create/Edit (включая reminders),
- [ ] Detail,
- [ ] Events/Main,
- [ ] Settings/More (где есть mutable state).
- [ ] Зафиксировать бэкап-регрессию отдельным прогоном тестов backup use cases.

Критерий завершения:
- [ ] Есть тестовая сетка, которая падает при удалении старого mutable-паттерна.

## 5.2 Инкремент B: Пилотная миграция одного экрана (RED -> GREEN -> REFACTOR)

Рекомендуемый пилот: `CreateEdit` (наиболее показательный, уже покрыт тестами).

- [ ] RED: добавить тесты на новый контракт иммутабельного UI-state для пилотного экрана.
- [ ] GREEN: заменить `MutableState<T>` в state-модели на plain-поля (`String`, `Boolean`, `Int?`, `LocalDate?` и т.д.).
- [ ] GREEN: перенести изменения полей в события ViewModel (`onTitleChanged`, `onDateChanged` и т.п.).
- [ ] GREEN: адаптировать Composable к новому контракту состояния.
- [ ] REFACTOR: удалить временные адаптеры, упростить расчет fingerprints/валидации.
- [ ] Обновить превью и связанные тесты.
- [ ] Прогнать: `make format`, таргетные тесты пилота.

Критерий завершения:
- [ ] Пилотный экран полностью работает на иммутабельном UI-state.
- [ ] Поведение reminders не изменилось.

- [ ] REFACTOR: удаление legacy-кода и дублирующих преобразований.

Приоритет миграции:

## 6. Риски и меры снижения

- [ ] Риск поломки сохранения состояния при повороте экрана.
Мера: тесты на recreation + `rememberSaveable` только в UI-слое.
- [ ] Риск изменения логики валидации/hasChanges.
Мера: отдельные unit-тесты до миграции и после.
- [ ] Риск скрытых регрессий в напоминаниях.
Мера: сохранять/редактировать reminder-сценарии в smoke-наборе каждого инкремента.
- [ ] Риск влияния на backup.
Мера: не трогать backup-модели; обязательный прогон backup-тестов на каждом релизном этапе.
