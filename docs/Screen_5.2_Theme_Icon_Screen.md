## Обзор

Theme and Icon Screen предназначен для настройки темы приложения и выбора иконки приложения. Экран позволяет пользователю выбрать между светлой, темной и системной темой, а также выбрать одну из доступных иконок приложения.

## Назначение

   - Заголовок: "Тема"

     - "Светлая"
     - "Тёмная"
     - "Системная"
   - Радиокнопки или Dropdown
3. **Секция "Иконка приложения"**
   - Заголовок: "Иконка"
   - Сетка с иконками (GridView)
   - Каждая иконка:
     - Превью иконки (65x65dp)
     - Галочка на выбранной иконке
   - При клике — установка иконки (если поддерживается системой)

## Стандарт Google по реализации темной темы (Material 3)

Согласно официальной документации Android для Jetpack Compose и Material 3, правильная реализация темной темы должна соответствовать следующим стандартам:

### Базовые требования

1. **Использование `isSystemInDarkTheme()`**
   - Это стандартный Composable-функция для отслеживания системной темы
   - Автоматически обновляется при изменении настроек устройства
   - Является дефолтным значением для параметра `darkTheme` в теме приложения
   - ✅ Реализовано в проекте (Theme.kt, строка 35)

2. **Использование стандартных компонентов Material 3**
   - `MaterialTheme` — обертка для применения темы
   - `darkColorScheme` — цветовая схема для темной темы
   - `lightColorScheme` — цветовая схема для светлой темы
   - ✅ Реализовано в проекте (Theme.kt, строки 10-31, 50-54)

3. **Интеграция с MainActivity**
   - Тема должна применяться на верхнем уровне в `setContent {}`
   - Все дочерние элементы автоматически наследуют тему
   - ✅ Реализовано в проекте (MainActivity.kt, строки 22-28)

### Динамические цвета (Material You)

Для Android 12+ (API 31+) Google рекомендует использовать динамические цвета (Material You):

1. **Официальные функции для динамических цветов:**

   ```kotlin
   import androidx.compose.material3.dynamicDarkColorScheme
   import androidx.compose.material3.dynamicLightColorScheme
   import androidx.compose.ui.platform.LocalContext

   @Composable
   fun jetpackDaysTheme(
       darkTheme: Boolean = isSystemInDarkTheme(),
       dynamicColor: Boolean = true,
       content: @Composable () -> Unit,
   ) {
       val colorScheme = when {
           dynamicColor && Build.VERSION.SDK_INT >= Build.VERSION_CODES.S -> {
               // Правильная реализация по стандарту Google
               if (darkTheme) dynamicDarkColorScheme(LocalContext.current)
               else dynamicLightColorScheme(LocalContext.current)
           }
           darkTheme -> DarkColorScheme
           else -> LightColorScheme
       }
       // ...
   }
   ```

2. **Преимущества динамических цветов:**
   - Автоматически адаптируются к персонализированным цветам устройства пользователя (обои, темы)
   - Обеспечивают согласованность с другими приложениями системы
   - Соответствуют принципам Material You (Material Design 3)
   - Улучшают пользовательский опыт за счет персонализации

3. **Требование контекста:**
   - Функции `dynamicDarkColorScheme(context)` и `dynamicLightColorScheme(context)` требуют контекст приложения
   - В Jetpack Compose контекст доступен через `LocalContext.current`

### Текущее состояние проекта

**Соответствует стандарту Google:**

- ✅ Использование `isSystemInDarkTheme()` для автоматического отслеживания системной темы
- ✅ Использование стандартных компонентов Material 3 (`MaterialTheme`, `darkColorScheme`, `lightColorScheme`)
- ✅ Интеграция с MainActivity на верхнем уровне
- ✅ Параметр `dynamicColor` для включения/отключения динамических цветов
- ✅ Проверка версии Android для включения динамических цветов только на поддерживаемых версиях

**НЕ соответствует стандарту Google (КРИТИЧЕСКИЕ ПРОБЛЕМЫ):**

- ❌ Не используются официальные функции `dynamicDarkColorScheme(context)` и `dynamicLightColorScheme(context)`
- ❌ При включенной опции `dynamicColor` игнорируется параметр `darkTheme` (баг в Theme.kt, строки 42-44)
- ❌ Отсутствует импорт `androidx.compose.ui.platform.LocalContext` для получения контекста
- ❌ Вместо использования официальных функций Material 3, используется фиксированный `LightColorScheme` для динамических цветов

### Необходимые исправления

Для соответствия стандарту Google требуется:

1. Добавить импорт `androidx.compose.ui.platform.LocalContext` в Theme.kt
2. Заменить фиксированный `LightColorScheme` на условие с использованием `dynamicDarkColorScheme(LocalContext.current)` и `dynamicLightColorScheme(LocalContext.current)`
3. Учесть параметр `darkTheme` при выборе динамической color scheme
4. Написать тесты для проверки правильного выбора цветовой схемы

## Примечания

1. **Стандарт Google по реализации темной темы (Material 3)**:
   - Приложение корректно использует `isSystemInDarkTheme()` для отслеживания системной темы (соответствует стандарту)
   - Используются стандартные компоненты Material 3: `MaterialTheme`, `darkColorScheme`, `lightColorScheme`
   - **КРИТИЧЕСКИЙ БАГ**: При включенной опции `dynamicColor` на Android 12+ приложение игнорирует параметр `darkTheme` и всегда использует `LightColorScheme`
   - Стандарт Google требует использования официальных функций для динамических цветов:
     - `dynamicDarkColorScheme(context)` — для темной темы
     - `dynamicLightColorScheme(context)` — для светлой темы
   - Эти функции обеспечивают интеграцию с Material You (персонализированные цвета устройства пользователя)
   - Текущая реализация не использует эти функции, что является отклонением от официальной документации Android

2. **Изменение иконки на Android**: Изменение иконки приложения на Android работает иначе, чем на iOS:
   - iOS использует `UIApplication.shared.setAlternateIconName()`
   - Android требует создания Activity Aliases в AndroidManifest.xml для каждого варианта иконки
   - Изменение иконки поддерживается только на Android 8.0 (API 26) и выше через `PackageManager.setComponentEnabledSetting()`
   - На более старых версиях Android функция должна быть скрыта или отключена

3. **Системная тема**: При выборе системной темы приложение должно следовать системным настройкам темы устройства (isSystemInDarkTheme()).

4. **Сохранение настроек**: Все настройки должны сохраняться в DataStore (Preferences DataStore) и применяться при следующем запуске приложения.

5. **Тестирование**: Все компоненты должны быть покрыты тестами перед использованием в других этапах (TDD подход).
